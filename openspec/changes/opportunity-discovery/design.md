## 上下文

当前项目是 React + TypeScript + Vite 前端，使用 React Router 的 hash 路由、Ant Design 组件、ECharts 图表和 Zustand 状态管理。工作台页面已经展示项目、风险、客户、文档和智能体任务，企业主数据位于 `src/mock/industry.ts`，投资项目数据位于 `src/mock/investment.ts`，企业详情页已有 `/companies/:companyId` 路由。

本变更不引入后端推荐服务，不改变产业洞察页面和顶部一级导航。需要新增一个“发现企业”页面，并让工作台的“推荐商机”承担快速入口职责。推荐结果需要能够继续进入企业画像、备选池和项目管理，形成从发现机会到新建项目的闭环。

## 目标 / 非目标

**目标：**

- 在工作台展示基于当前用户投资风格生成的 3~5 家推荐企业。
- 新增“发现企业”路由，提供完整企业列表、推荐解释和多维组合筛选。
- 将生命周期阶段为“投”和“管”的投资项目企业作为投资风格样本。
- 支持政府奖励、认定认证、科技类认证、产业链、区域、企业规模、投融资、经营风险、技术人才等分组筛选。
- 让企业名称、企业画像、加入备选池和新建项目动作可以连续衔接。
- 使用可解释的前端 mock 规则支撑演示，并为后续替换真实推荐接口保留数据边界。

**非目标：**

- 不修改产业洞察原有的产业链图谱、产业分析和企业分布主流程。
- 不新增顶部一级菜单，不改变现有项目、客户、风险、文档和智能体路由命名。
- 不实现真实外部企业数据采集、机器学习模型或在线推荐服务。
- 不在用户点击推荐卡片时自动创建项目，项目创建必须由用户确认。

## 决策

### 1. 工作台入口和发现企业路由

在 `Dashboard` 增加“推荐商机”模块，展示 3~5 张紧凑的企业推荐卡片。卡片显示企业名称、产业链、推荐度、匹配标签和一条推荐理由，并提供“查看画像”和“更多”操作。

“更多”进入新增的 `/opportunity-discovery` 路由，页面标题为“发现企业”。企业名称和“查看画像”复用现有 `/companies/:companyId` 路由，避免复制企业详情逻辑。

选择该路由名称而不是复用产业洞察路由，是为了区分“产业研究”和“投资机会发现”：产业洞察负责发现产业方向，发现企业负责基于投资策略筛选具体企业。

### 2. 投资风格样本与推荐结果

推荐计算使用纯函数从现有投资项目生成 `InvestmentStyleProfile`：

- 样本项目：`project.stage === '投' || project.stage === '管'`。
- 产业偏好：聚合样本企业的 `industries`。
- 企业特征偏好：聚合样本企业的 `tags`、`mainBusiness` 和综合/科创评价。
- 区域偏好：从样本企业地址提取区域信息；演示数据缺少标准区域字段时使用现有地址文本和补充 mock 元数据。
- 排除项：已纳入投中或投后样本的企业不再次作为新的推荐候选。

推荐流程分为两步：先应用用户选择的筛选条件作为硬条件过滤候选企业，再按匹配度排序。匹配度由产业链匹配、标签和技术方向匹配、区域匹配、综合评价、风险状况和投融资特征组成。每条结果同时生成 `recommendationSources`、`matchedDimensions` 和 `reason`，使列表可以解释“为什么推荐”。

如果没有可用的投中/投后样本，使用南通重点产业、产业链匹配和企业综合评价作为演示降级规则，并在界面标注为“基于重点产业的推荐”，避免出现空白页面。

### 3. 多维筛选模型

新增机会筛选状态由 Zustand 管理，至少包含当前推荐策略、关键词、已应用筛选条件、筛选结果数量和候选列表。筛选条件使用分组结构，便于后续接入服务端枚举：

```text
政府奖励
认定认证
科技类认证
产业链与行业
区域
企业规模与发展阶段
投融资特征
经营与风险
技术与人才
```

每个分组包含多选项和“全部”选项，支持展开/收起。页面提供已选条件摘要、结果数量、确认查询和清空条件。确认查询后统一更新已应用条件，避免用户勾选过程导致列表频繁跳动。

推荐策略与筛选条件分离：推荐策略决定排序依据，筛选条件决定候选范围。策略至少包括“投资风格相似”“产业链补链”“南通重点产业”“高成长优选”和“综合推荐”。用户可以先选择策略，再叠加多个筛选分组。

### 4. 页面组件与交互

发现企业页采用“顶部策略区 + 多维筛选区 + 企业结果表格”的结构：

- 策略区：展示当前投资风格来源、已使用的投中/投后项目数量、推荐策略切换和搜索框。
- 筛选区：使用 Ant Design 的 `Collapse`、`Checkbox.Group`、`Tag` 和 `Button`，以分组折叠方式呈现大量选项。
- 结果区：使用 `Table` 展示推荐度、企业名称、推荐来源、产业链、所在地、认证标签、综合评分、风险等级、推荐理由和操作。
- 企业名称使用链接或按钮进入企业详情；推荐理由使用可换行文本，避免长文本破坏表格布局。
- “加入备选池”沿用企业详情页已有交互；“新建项目”携带企业 ID、推荐来源和推荐理由进入项目管理，并打开新建项目表单等待用户确认。

### 5. 路由状态与闭环衔接

发现企业页到项目管理的跳转使用 URL 查询参数传递推荐上下文，例如企业 ID、推荐来源和推荐理由。项目管理读取参数后预填企业和推荐信息，但仍由用户点击提交才创建项目。企业详情页、项目详情页和智能体页面继续使用已有路由，不新增重复详情页。

推荐上下文进入项目后写入项目的演示数据结构，用于后续项目详情显示“机会来源”和“推荐依据”，并为尽调、估值、合同审核和投后监测提供上下文。

## 风险 / 权衡

- [现有企业 mock 数据的认证、区域、融资和人才字段不完整] -> 增加前端演示用的推荐元数据映射，集中放在机会推荐数据模块；真实接口接入时替换数据源，不把字段散落到页面组件。
- [投中/投后样本数量较少导致推荐结果单一] -> 使用产业链、南通重点产业和综合评价作为降级推荐规则，并展示推荐来源。
- [筛选维度过多导致页面过长] -> 分组折叠、默认只展开常用分组，并在顶部展示已选条件和结果数量。
- [用户不理解推荐结果] -> 每行提供推荐来源、匹配维度和自然语言推荐理由；推荐度只作为排序参考，不代替投资决策。
- [推荐结果与已有项目重复] -> 默认排除投资风格样本企业和已有项目企业，并提供明确的排除状态。
- [推荐卡片直接创建项目可能造成误操作] -> 推荐卡片只允许查看画像或进入发现企业，项目创建必须在详情或列表中二次确认。
- [前端 mock 推荐规则与未来服务端结果存在差异] -> 将候选过滤、评分和推荐理由封装为独立接口形状，后续可以替换为异步 API，不改变页面契约。

## 迁移计划

本变更不涉及数据库迁移和生产数据迁移。实现时按以下顺序接入：先新增推荐类型和 Zustand store，再新增发现企业路由和列表页，随后在工作台接入推荐卡片，最后打通企业详情到项目管理的推荐上下文。若需要回滚，移除工作台入口和发现企业路由即可，不影响现有产业洞察、企业详情和项目管理页面。

## 待确认事项

- 真实环境中的政府奖励、认证和科技类标签枚举需要以后端企业库字典为准；当前仅定义前端演示分组和选项。
- 推荐度的具体权重当前只服务演示排序，后续接入真实推荐模型时需要由业务方确认权重解释和审计要求。
